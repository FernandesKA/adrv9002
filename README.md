# ADRV9002 Linux Kernel Driver

## Для чего нужен этот драйвер:

- Управляет трансивером ADRV9002 по SPI с HAL-абстракцией для SPI/GPIO reset.
- Предоставляет IOCTL API (инициализация, частоты TX/RX, усиление, вкл/выкл каналов, МШУ, аппаратный reset).

## Сборка

- Внутренний уровень ADI-логов при сборке: `ADI_COMPILED_LOGLEVEL=ADI_LOGLEVEL_WARN` (ниже — не компилируется).

## Логирование

Есть два независимых уровня логов:

- Verbose (dmesg): человекочитаемые сообщения через `dev_info()` при настройках и управлении.
  - Включение: `modprobe adrv9002 verbose_logs=1` или `echo 1 > /sys/module/adrv9002/parameters/verbose_logs`
  - Пример (успешная перестройка частоты):
    - `[... ] adrv9002 spi1.0: Frequency set ch0: 1000000000 Hz (15234 us)`

- Detailed (кольцевой буфер): подробная трассировка шагов перестройки и состояний канала.
  - Хранится в отдельном кольцевом буфере, доступном по `/proc/adrv9002_log`
  - Включение: `modprobe adrv9002 detailed_logs=1` или `echo 1 > /sys/module/adrv9002/parameters/detailed_logs`
  - По умолчанию выключено, чтобы не влиять на производительность
  - При первой фатальной ARM-ошибке буфер «замораживается» (freeze), чтобы сохранить контекст
  - После первой авто-реинициализации (если включена) детальные логи автоматически отключаются
  - Статус буфера и его заполнение можно посмотреть в `/proc/adrv9002` (раздел Detailed Logs)

Отдельный статус-файл `/proc/adrv9002` содержит сводку по устройству, параметрам и (при включенном профилировании) время выполнения IOCTL.

## Автоматическая реинициализация при ошибках

- **По умолчанию выключено**: драйвер может автоматически переинициализировать трансивер при некоторых ошибках ADI API.
- Функция изначально предназначалась как workaround для нестабильности API, но сейчас рекомендуется держать выключенной.
- Включение: `modprobe adrv9002 auto_reinit=1` или `echo 1 > /sys/module/adrv9002/parameters/auto_reinit`.
- Счетчик автоматических реинициализаций доступен в `/proc/adrv9002` (поле `auto_reinit_count`).
- При реинициализации используются сохраненные буферы профиля и потока из первичной инициализации.

## Настройка таймаутов перестройки частоты

Операция `ADRV9002_IOC_SET_FREQ` требует ожидания смены состояния канала. Оптимизации:

1. **Базовая задержка опроса** настраивается параметром `freq_change_timeout_us` (по умолчанию 200 мкс):

  ### Более быстрый отклик (агрессивный опрос)
  echo 200 > /sys/module/adrv9002/parameters/freq_change_timeout_us
   
  ### Более консервативно (если возникают ошибки переходов состояний)
  echo 1000 > /sys/module/adrv9002/parameters/freq_change_timeout_us

## Измерение времени

- Для профилирования времени выполнения IOCTL операций используйте параметр `measure_timing`.
- Включение: `modprobe adrv9002 measure_timing=1` или `echo 1 > /sys/module/adrv9002/parameters/measure_timing`.
- При включённом параметре измеряется время выполнения каждого IOCTL вызова от точки входа до выхода.
- Последние измеренные значения доступны в `/proc/adrv9002`:
  - `ioctl_init_time_us` — время выполнения ADRV9002_IOC_INIT (инициализация)
  - `ioctl_set_freq_time_us` — время выполнения ADRV9002_IOC_SET_FREQ (установка частоты)
  - `ioctl_get_freq_time_us` — время выполнения ADRV9002_IOC_GET_FREQ (чтение частоты)
  - `ioctl_set_gain_time_us` — время выполнения ADRV9002_IOC_SET_GAIN (установка усиления)
  - `ioctl_rx_enable_time_us` — время выполнения ADRV9002_IOC_RX_ENABLE (управление RX)
  - `ioctl_tx_enable_time_us` — время выполнения ADRV9002_IOC_TX_ENABLE (управление TX)
  - `ioctl_reset_time_us` — время выполнения ADRV9002_IOC_RESET (сброс устройства)
  - `ioctl_lna_enable_time_us` — время выполнения ADRV9002_IOC_LNA_ENABLE (включение LNA)
  - `ioctl_lna_disable_time_us` — время выполнения ADRV9002_IOC_LNA_DISABLE (выключение LNA)

Пример использования:

### Включить измерение времени
echo 1 > /sys/module/adrv9002/parameters/measure_timing

### Посмотреть результаты
cat /proc/adrv9002 | grep ioctl

## Параметры модуля

- `verbose_logs` (bool): включает информативные сообщения в dmesg для операций настройки. По умолчанию `0`.
- `detailed_logs` (bool): включает подробную трассировку в кольцевой буфер `/proc/adrv9002_log`. По умолчанию `0`.
- `measure_timing` (bool): включает профилирование времени IOCTL-операций; результаты видны в `/proc/adrv9002`. По умолчанию `0`.
- `freq_change_timeout_us` (int): базовый интервал сна (в микросекундах) для опроса состояний при перестройке частоты; меньше — быстрее. По умолчанию `200`.
- `auto_reinit` (bool): авто-реинициализация устройства при неожиданных ошибках IOCTL с повтором операции. По умолчанию `0` (выключено).

## IOCTL команды 

- ADRV9002_IOC_INIT: загрузка профиля/stream.
- ADRV9002_IOC_SET_FREQ / ADRV9002_IOC_GET_FREQ: частота TX/RX (Hz, порт/канал).
- ADRV9002_IOC_SET_GAIN: установка усиления (порт/канал).
- ADRV9002_IOC_RX_ENABLE / ADRV9002_IOC_TX_ENABLE: включение/выключение канала.
- ADRV9002_IOC_RESET: аппаратный reset чипа.
- ADRV9002_IOC_LNA_ENABLE / ADRV9002_IOC_LNA_DISABLE: управление МШУ по битовой маске (no-op, если МШУ отсутствует на плате и в dts).

## МШУ (LNA)

- DT: свойство amp-gpios = <...> перечисляет линии в порядке битов маски.
- Включение/выключение: используйте ADRV9002_IOC_LNA_ENABLE/ADRV9002_IOC_LNA_DISABLE с маской (бит 0 — первый GPIO и т.д.).
- Если amp-gpios не задан или GPIO нельзя сконфигурировать, драйвер считает МШУ отсутствующим: ioctl возвращают 0 без изменения линий.

## Reset: два варианта

- GPIO: укажите reset-gpios в dts
- Регистровый PL: пропишите reset-reg (addr, mask, reset, unreset).
- Если reset не задан ни через GPIO, ни через reset-reg, драйвер загружается без управления линией сброса; ответственность за первоначальный reset остаётся на загрузчике/FPGA.
- Без выделенной линии reset повторная инициализация после сбоя может не сработать: adrv9002 может остаться в неизвестном состоянии.

## Статус в procfs

- После загрузки модуля статус доступен в /proc/adrv9002.
- Поля: driver_version, initialized, auto_reinit_count, lna_present, lna_gpios, lna_state, lo_freq[0/1], gain[0/1].
- При включённом параметре `measure_timing` также доступны времена выполнения всех IOCTL операций (в микросекундах).
